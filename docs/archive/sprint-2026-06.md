- Runner mix: codex 30, gemini 21, claude 7, fake 2 — but zero completed live Claude task runs (2 smokes, 4 manual records, 1 auth failure `20260610T171639Z-770f1bfb`).

- `task_kind` says `docs` on 30/60 runs; most of those are code implementation work.

## What the Data Says

1. **The session-capture leak is the biggest hole in the loop.** 24 of 28 feedback reports have `run_id: none`. The largest bodies of real work (idiolect R1–R11, 189 passing tests; most of the moshpit-labs 06-10/06-11 ship stream; growth-run missions) and the worst failures (the destructive shared-worktree race in `report-20260609T072839Z-8df89ea6`) exist only as prose reports. The ledger has no record of its own most important evidence.
2. **Labels actively mislead routing.** `provider_needs_input` over-fires: 4 blocked runs (`20260610T100302Z-c99e1c12`, `20260610T103519Z-6f1cf160`, `20260610T050809Z-596aef0a`, `20260611T025704Z-a4f09df3`) changed files, passed checks, and 3 were later verdict-accepted — deflating codex success stats. Checks pass vacuously on runs that did nothing (`20260610T171639Z-770f1bfb` recorded 158/158 npm-test pass after Claude failed to log in). Write-scope precedence is inconsistent (violations count 3 by status_reason, 4 by `write_scope.violations`).
3. **Provider metadata observation is 0%.** Codex emits 733 B – 2.4 MB of stderr per run (median ~79 KB) that Rux stores only as a byte count (`20260607T180353Z-747ebaa4`: 76,627 bytes). Cost-aware or model-aware routing is impossible today.
4. **Shared-worktree execution contaminates evidence.** Changed-file lists include pre-existing dirty entries; untracked directories collapse (`20260609T072128Z-3b5c3b88` recorded `playbooks/` instead of the file); a concurrent session's untracked dir failed a release-gating smoke (`20260611T093820Z-c85e995a`). Two repos hit destructive races before the dirty-tree block shipped.
5. **Early routing signal exists, but it is narrow and partly unstructured.** Codex respected prompt-level scope 3/3 on scoped mechanical tasks in idiolect (`20260609T064516Z-0bdc2dd7`, `…d59dcad0`, `…aeacf58d` — scope was prose in the prompt, not a recorded `write_scope`). Gemini was fast for doc authoring (39–46s vs codex 127–180s in growth-run) but violated write scope in two repos (moshpit-labs `20260610T051655Z-d5a35b16`, idiolect `20260609T064802Z-b555e920`) and cannot self-run checks (`run_shell_command` unavailable). Two of the four recorded write-scope violations are codex, so "codex safe / gemini unsafe" is a hypothesis to test, not a conclusion. These observations become heuristics only if the relabeled evidence still supports them.

The wave is three sprints on the weekly release train. Sprints 2.2 and 2.3 contain minor-version changes (new ledger fields, output behavior); they are not patch-train items. ## Sprint 2.1 — Close the Capture Leak

The session agent (Claude Code, Codex session, etc.) does most real work and is invisible to the ledger. Fix attribution first, then linkage.

1. **Session baseline for `rux record`.** Add `rux record --start`, which appends a session-baseline event (id, HEAD, dirty-file snapshot) to the ledger; a later `rux record` diffs against it (default: most recent open baseline, with a printed warning when ambiguous). Files already dirty at baseline that change again during the session are reported but flagged as contaminated. No new top-level noun; this stays under `record`. Acceptance: a baselined record's `changed_files` equals files changed since the baseline (smoke-tested); `20260610T100558Z-d3577cf2`-style identical before/after listings become impossible for baselined records.
2. **Report→run linking, keyed by kind.** `rux report --kind success` (work that shipped) requires one of `--run-id`, `--record` (mint a baselined manual record inline), or `--no-run` with a reason. All other kinds (bug/ux/adapter/docs/install/idea/other) stay frictionless with a stderr nudge when unlinked — RUX-007's low-friction learning is preserved. Acceptance: smoke test that `report --kind success` without linkage exits non-zero and `--kind ux` still succeeds with a warning.
3. **Changed-file attribution everywhere.** Compute `changed_files` against a per-run baseline; expand untracked directories to file lists. Acceptance: no run records a directory as a changed file; dirty-tree contamination (`20260607T122055Z-fa871436`) cannot recur on new records.
4. **Stretch — passive observation prototype.** Ingest completed provider session transcripts (e.g. Claude Code project logs) as ledger candidates confirmed with one command. Interface Level 1 (Observe) from the vision. Ingested records follow the import rules: `confidence: low`, excluded from routing evidence and release gates by default, upgradeable only via checks or verdicts; they do not count toward the live-Claude dogfood quota. No silent imports.

## Sprint 2.2 — Fix the Labels, Observe the Metadata

Routing reads labels; today they lie in both directions. Two release trains: 2.2a is the classification layer, 2.2b is adapters and taxonomy. Provider-text extraction (2.2b-1) should land with or before the classifier rewrite (2.2a-2), so heuristics are written against extracted text, not raw mixed output.

**2.2a — classification layer (one train):**

1. **Deterministic signal precedence.** `write_scope_violation` > `provider_plan_changed_files` > `check_failed` > `provider_needs_input` > `completed`. Plan-mode safety stays above needs-input — a plan-mode run that edited files and ended with a question must remain a safety failure. Mostly a small fix near the `status === 'ok'` guard (`src/cli.mjs:1211`) plus regression tests.
2. **Tighten `needs_input` and add read-time reclassification.** A run that changed files and passed its check should not classify as blocked because its closing summary contained a question. Existing ledgers are never rewritten: compute a derived `effective_status_reason` + `classifier_version` at read time, shown on all read surfaces next to the stored value; routing consumes only the effective value. Acceptance: vendor sanitized fixture JSONL of the 4 known false positives plus the `770f1bfb` auth-failure record into `tests/fixtures/`; smoke asserts the reclassifier output over those fixtures, and validates the new heuristic against all historical blocked runs, not just the known four.
3. **Vacuous checks.** A check is `vacuous` when it passed but the run did nothing: check passed AND (run status not ok, OR change-like task_kind with zero changed files). Review/plan tasks with legitimately empty diffs are not penalized. Implemented at the eval/routing layer for historical records (read-time exclusion) and stamped on new check events. Acceptance: fixture of a `770f1bfb`-shaped record is excluded from routing evidence.
4. **Retire the dead inline `human_verdict` field.** The verdict join already exists in `show`/`eval`/`export`; the task is small: stop writing `human_verdict: null` on new records, keep readers tolerant of old records, add a verdict column to `ls`.

**2.2b — adapters and taxonomy (one train):**

5. **Record mode and purpose as first-class fields.** (a) Persist `provider_mode` (`plan|write|internal`) as a top-level run field instead of forcing argv parsing — same meaning as the existing flag, no new noun. (b) Add `rux run --purpose probe`; probes are excluded from `rank`/`suggest` by default (`--include-probes` to override). Justification: consumers stop parsing argv, and probes stop polluting live-task stats (growth-run probe `20260609T071629Z-af1ebb28` is recorded as a live docs task today). Acceptance: a probe run never appears in suggest evidence counts.
6. **Honest `task_kind`.** No interactive prompts (headless callers would hang). Absent `--task-kind`, record `task_kind: unspecified` — never inferred-as-docs — with a stderr nudge, plus `task_kind_source: user|unspecified` and a post-run `task_kind_suggestion` derived from changed-file extensions, recorded as a suggestion, never an auto-rewrite. Acceptance: new runs stop defaulting to `docs` for code work.
7. **Provider output parsing.** Verified against installed CLIs: claude supports `--output-format json` (includes `total_cost_usd`), codex supports `exec --json` (JSONL events), gemini supports `--output-format json`. Parse structured output; extract assistant text and feed it to the classifiers; populate model/effort/cost with `metadata_sources: observed`; persist a parsed extract rather than a byte count; record model fallback structurally; fall back gracefully to byte-count capture when structured mode is absent. Explicit subtask: rewrite the mock provider CLIs to emit representative JSON so the smoke suite covers the parsed path. Acceptance: smoke asserts `metadata_sources.model === "observed"` against the JSON-emitting mocks; each adapter change lands with a provider or mock smoke; flags re-verified against installed CLI versions at implementation time.

## Sprint 2.3 — Surfaces and Labeling Friction

Only after the record is trustworthy do the surfaces deserve attention.

1. **Human-readable output on TTY.** When stdout is a TTY, print readable output; when piped, keep JSON — the current script contract (README: "`rux run` keeps stdout as JSON for scripts") stays intact for pipes, and `--json` forces it. This amends sticky decision RUX-003 and requires an explicit `docs/DECISIONS.md` amendment before implementation; it ships as a minor release with smoke coverage for both paths. `status` leads with the top-ranked evidence statements generated from `rank`/`eval` at read time for this repo's ledger, each citing run IDs and evidence counts — never canned claims, never cross-repo aggregation (the ledger is repo-local).
2. **Verdict and mark friction.** Prompt for a verdict at run completion only when stdin and stdout are TTYs; never when piped. Headless path: `--verdict <v>` on `rux check`/`rux record`. Session-end verdicts are the host agent's ritual per AGENTS.md, not a Rux process (no daemon). The `rux mark` nudge is computed at `record`/`run`/`status` time by diffing changed-file overlap with recent runs — no watcher. Acceptance: smoke asserts the headless verdict path writes a verdict event; piped invocations never prompt.
3. **Smoke stability without dishonesty.** Record every smoke attempt as its own ledger event with attempt linkage; retries happen at the orchestration level and release evidence cites the attempt chain, not just the final pass — silently retried release gates would be exactly the vacuous-pass dishonesty Sprint 2.2 kills. Diagnose timeouts (the two 120s codex timeouts `20260611T093303Z-4906853a`, `20260611T093555Z-5ca0cc5e` recorded zero diagnostic output) and rely on the dirty-tree guard, not retries, for concurrent-session false positives (`20260611T093820Z-c85e995a`).
4. **README rewrite.** Lead with the five-minute experience: 3-command quickstart, one terminal recording, then the vision links. Keep the `check:docs` contract intact.
5. **Landing page (gated).** One static page: the one-question pitch, a terminal recording of the loop the ledger can already back — `run → show → verdict → outcome` — with `suggest` shown only as real, unedited output including its maturity label and evidence counts. Ships only after the wave exit criteria pass, and is the first item cut if the wave slips. The page lives on claims the ledger can demonstrate; nothing more.

Deferred from this wave: `--help` command grouping (cosmetic, does not feed the sprint goal).

## Dogfood Quotas During the Wave

The wave fails without new labeled evidence. Alongside the code:

- Every substantial session ends with a baselined `rux record` and a verdict. No more `run_id: none` success reports.
- Log in to Claude CLI and capture at least 5 completed live Claude task runs — the runner currently has zero. This quota is at risk behind the auth flow that already failed once; treat login as a day-one task.
- Diversify task kinds: at least 5 honest `bugfix` and 5 `review` runs (today: 1 bugfix, 4 review).
- Exercise lifecycle marks on real events (at least one revert and one accepted-downstream; today: zero marks ever).
- Once read-time reclassification ships, triage the 34 unverdicted historical runs: verdict what is still judgeable, leave the rest unlabeled.
- Usage targets (tracked here, not as CI gates): model observed on the majority of new live runs for at least one provider; new-run verdict coverage above 60%.

## Anti-Scope

Unchanged from the guardrails, restated because the data tempts otherwise:

- No dashboard or web UI; the CLI loop is not proven yet.
- No hosted sync, accounts, or team SaaS.
- No daemon, watcher, or background process — all nudges compute at command time.
- No dynamic rosters or learned routing beyond what the (now honest) evidence supports.
- No LLM-judge outcome labels; checks and human verdicts remain the only labels.
- No rewriting of existing ledger records; all label fixes are new-record fields or read-time views with a `classifier_version`.
- No cross-repo aggregation surfaces; the ledger stays repo-local this wave.
- Landing page is one static page, gated on wave exit, not a marketing site.

## Wave Exit Criteria

- `rux suggest` can cite at least two defensible, evidence-backed heuristics at maturity `directional` or better. Heuristic content is an output of the relabeled evidence, not a target — the two observations in "What the Data Says" §5 are hypotheses to confirm or kill. A third (Claude-related) heuristic is desirable but at risk behind the login dependency.
- Zero `run_id: none` success reports in new dogfood data.
- Model metadata observed (not user-typed) on the majority of new live runs for at least one provider.
- No known mislabeled statuses or vacuous passing checks among new records; historical records carry honest effective labels on all read surfaces.
- At least one real lifecycle mark recorded.
- `SCHEMA_VERSION` 2 shipped with v1-tolerance fixture tests.
- README (and landing page, if shipped) describe only what the ledger can demonstrate.
